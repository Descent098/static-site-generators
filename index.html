Static site generators

An overview of markup transformation & Static Site Generator (SSG) demo

What is a static site generator?

A static site generator is a system that allows you to create a website using a template and a set of data. You can think about this as translating/transpiling data from a language or languages to a website. Universally SSG's have 3 parts:

  • The source/state/content files (or sometimes a single file)
  • The template file(s)
  • The output file(s)

What are the alternatives?

  • Static (raw html/css/js files)
  • Dynamic (i.e. Wordpress, squarespace etc.)
  • SSG's (right in the middle)

Why use a SSG?

  • Easier to maintain than a static site
  • More configurable than a static site
  • cheaper and less complex than a dynamic site (i.e. wordpress)

Principles of an SSG

There are some common principles that static site generators use that are important to be aware of

State driven templating

This is the concept that unlike a typical 'static' site there is some sort of state used in conjunction with predefined templates to drive generating the output files

Source files

Source files are typically human-readable files (though don't have to be) and are used to drive the state of the template files. They are often Markdown or rich source text files, along with some sort of configuration file in JSON, YAML or TOML

Template files

These files are typically HTML-like files that use templating languages to pull data from the source files and generate output files. This is also an example of Code generation

Output files

These files are the result at the end that can be hosted somewhere. These will typically be HTML/CSS/JS files along with any additional static files like images.

Markup transformation

Lost of SSG's are built on the principle that markup systems (like HTML) are used to express common structures and content. With this the specifics of one system are often analogous to another which means you can convert them 1 to 1 from one form to another (i.e. markdown to HTML)

Hugo overview

Hugo is one of the most popular static site generators that is sed today

Everything is sections

Hugo like ezcv uses sections to split up content, each section has the content for that section and the template(s). Everytime you create a new section there will be a folder inside layouts with the same name, one in content with the same name, and ideally a file in /archetypes with the same name

Source files

Hugo uses markdown files for content and TOML files for configuration

Archetypes

Archetype files essentially are CONTENT templates that can be used to make sure your markdown files have the correct metadata

Why use Archetypes?

So for example if you have a section called events that you need a date and an image for, you can write an archetype file to enforce that, which will also use those defaults if people create content at the command line

Example

So if this was in archetypes/events and someone ran the code below it would setup the file for them

hugo new events/event.md

Template files

Hugo does not have a seperate file type for template or output files, both are .html files and hugo enforces a specific file structure that I will explain here

Template Scope

Hugo silo's access to variables based on scope. There are a lot of rules around this, but when in doubt pass . to a template which will pass the whole site context to the tempalte (this will be important for partials later)

Hugo scope

/static

This all of the static files such as css, javascript, fonts and images

Accessing /static files

Let's say you have a file in /static/css/main.css to access it in templates use /css/main.css as the path

/_default

This folder contains all of the 'shells' where things that don't change across pages go (i.e. navbars, stylesheet imports, footers etc.)

/_default/baseof.html

This is used as a base template that is invoked when using any files (unless other defaults are specified)

/_default/list.html

This is used as the base template that is invoked when using any list.html files

/_default/single.html

This is used as the base template that is invoked when using any single.html files

sections

This would be a folder with the section name and will house the templates for that section. These are not required if your files in /_default also work for what you need

/section/list.html

This file is used for an 'overview' page. For example if you had a section with a bunch of events or blog posts this page would be what you use to show off a LIST of all of them. This also creates a route of /section

/section/single.html

This file is what will render individual (SINGLE) pages of content. So this would be the template used to render a page for EACH blog post or EACH event

Partials

Partials are essentially re-usable snippets that you can embed in other tempaltes. They are very handy for components you will use in various different places like a custom themed blog-post feed. These also work off the code generation mentioned earlier

Partials

You can also pass variables to partials to allow them to be used in different circumstances

/Themes

This folder will allow you to put all the content for the templates inside a subdirectory for easy code sharing. Once you have the theme code inside the folder you just need to add the following line to config.toml

theme = 'theme_name'

Output files

Hugo will output HTML/CSS/jS files
